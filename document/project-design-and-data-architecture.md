# B.CY Library 기획 의도와 구현 정리

## 1) 간단 정리

### 왜 만들게 되었는가
- 읽은 책 목록을 빠르게 기록하고, 다시 빌릴 때 참고할 수 있는 개인용 도서 기록 도구가 필요했다.
- 주 사용 환경을 모바일 크롬(PWA 설치)으로 잡았고, 초보 사용자도 쉽게 쓰는 것을 우선 목표로 삼았다.
- 서버 운영 부담 없이 안정적으로 배포하기 위해 정적 호스팅(GitHub Pages) 중심으로 설계했다.

### 어떻게 만들었는가
- 프론트엔드는 순수 JS 기반의 PWA 구조로 구현했다.
- 오프라인/로컬 중심 사용성을 위해 브라우저 IndexedDB를 기본 저장소로 사용했다.
- 도서 검색은 Google/OpenLibrary/Naver를 조합했고, Naver는 브라우저 직접 호출 제약을 우회하기 위해 Cloudflare Worker 프록시를 사용했다.
- 데이터 백업/복원은 JSON 파일 내보내기/가져오기 + GitHub Gist 동기화 방식으로 보완했다.

### 어떤 시행착오가 있었고 어떻게 해결했는가
- 같은 기기인데도 데이터가 사라진 것처럼 보이는 문제
- 원인: IndexedDB는 Origin(주소) 단위로 분리 저장된다.
- 해결: README에 주소별 저장 분리 안내를 명시하고, 동일 주소 접속을 사용하도록 운영 가이드를 추가했다.

- Naver API를 브라우저에서 직접 호출할 때 제한(CORS/보안 구조) 문제
- 원인: 클라이언트에서 직접 비밀키를 노출하거나 호출 제약에 걸릴 수 있다.
- 해결: Cloudflare Worker를 프록시로 두고, 브라우저는 Worker만 호출하도록 분리했다.

- 백업 가져오기 시 중복 데이터 유입 가능성
- 원인: 과거 데이터 형식에는 guid가 없을 수 있어 단순 병합 시 중복 가능성이 있었다.
- 해결: guid 기반 중복 방지 + 구버전 호환용 약한 키(title+authors+finishedAt) 비교를 함께 적용했다.

- 기존 데이터에 guid가 없는 상태에서 확장 기능 적용 어려움
- 원인: 초기 저장본에 guid가 비어 있을 수 있었다.
- 해결: 앱 시작 시 일괄 보정 마이그레이션(ensureBookGuids)으로 누락 guid를 채우고 갱신 시간을 업데이트했다.

### DB 서버가 없을 때 저장/조회 방식
- 기본 데이터 저장은 브라우저 IndexedDB의 books 오브젝트 스토어를 사용한다.
- 조회는 updatedAt 인덱스를 사용해 최신순 커서 스캔으로 리스트를 만든다.
- 외부 백업은 JSON 파일 또는 GitHub Gist를 사용하고, 앱 복구 시 가져오기 로직으로 재적재한다.

## 2) 기술 상세

### 2-1. 저장소 구조(IndexedDB)
- 데이터베이스 이름: bcy-library
- 버전: 3
- 스토어: books (keyPath=id, autoIncrement)
- 주요 인덱스: title, authors, updatedAt, finishedAt, category, guid

핵심 설계 포인트:
- 저장은 id 자동 증가 키를 사용해 단순 CRUD를 안정화했다.
- 동기화/중복판별용으로 guid를 별도 필드로 관리했다.
- 목록 조회 성능을 위해 updatedAt 인덱스로 정렬 비용을 줄였다.

### 2-2. CRUD와 데이터 정규화
- 추가(addBook): title/authors/note trim, createdAt/updatedAt ISO 시간 저장, guid 생성
- 조회(getBook): id 단건 조회
- 수정(updateBook): 기존 레코드 병합 후 updatedAt 갱신, guid 누락 시 보강
- 삭제(deleteBook): id 기반 삭제
- 목록(listBooks): updatedAt 인덱스 커서를 prev 방향으로 순회해 최신순 반환

이 방식의 장점:
- 서버 없이도 반응성이 좋고, 네트워크 불안정과 무관하게 동작한다.
- 클라이언트 단에서 즉시 저장/조회가 가능해 입력 UX가 빠르다.

### 2-3. 가져오기/내보내기와 중복 방지
- 내보내기(export): 현재 목록을 app/version/exportedAt/books 구조의 JSON으로 다운로드
- 가져오기(import): 기존 데이터는 유지하고 추가만 수행

중복 방지 로직:
- 1차: guid가 있으면 guid Set으로 중복 검사
- 2차(구버전 호환): guid가 없는 데이터는 title+authors+finishedAt 정규화 조합(weak key)으로 중복 검사
- 결과: 신규만 insertImportedBook으로 추가, 중복/오류 건은 건너뜀

### 2-4. 백업 계층(로컬 + 원격)
- 로컬 백업: 파일 다운로드(JSON)
- 원격 백업: GitHub Gist

Gist 백업 흐름:
- 토큰 검증 후 설정(localStorage) 저장
- Gist ID가 없으면 생성, 있으면 PATCH 업데이트
- 404 등 예외 시 재생성 경로를 둬 백업 복구력을 확보

주의점:
- 토큰은 브라우저 localStorage에 저장되므로, 개인 기기 사용과 권한 최소화(gist scope)가 전제된다.

### 2-5. 서버 없는 아키텍처에서의 API 처리
- 검색 데이터는 외부 API를 호출해 화면에 반영한다.
- Naver API는 브라우저 직접 호출 대신 Worker 프록시 경유로 호출한다.
- 프록시를 쓰면 클라이언트 단에 민감한 인증정보를 직접 싣지 않고 호출 경로를 통제할 수 있다.

### 2-6. 서버리스/로컬 퍼스트 설계의 의미
- 별도 DB 서버 운영이 없어도 개인 기록 앱 요구사항(입력, 조회, 검색, 백업)을 충족한다.
- 배포 복잡도를 낮추고 운영비를 줄일 수 있다.
- 데이터 일관성은 단일 기기 로컬 저장소 + 명시적 백업 절차로 관리한다.

## 3) 한 줄 결론
- 이 프로젝트는 서버 없는 PWA를 전제로, IndexedDB를 핵심 저장소로 삼고 JSON/Gist 백업 계층을 추가해 실사용성과 복구성을 동시에 확보한 구조다.
